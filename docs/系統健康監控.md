# 系統健康監控

## 文件目的

說明本專案的健康檢查端點與人工巡檢頁面，讓維運人員可以快速判斷網站、API、資料庫、日誌、身分驗證、檔案系統、主機資源與安全設定是否正常。

---

## 1. 部署探針

系統保留 ASP.NET Core Health Checks 端點：

| 路徑 | 用途 | 說明 |
|------|------|------|
| `/health/live` | Liveness | 檢查應用程式處理程序是否仍存活。 |
| `/health/ready` | Readiness | 檢查資料庫連線是否可用，供部署平台判斷是否可接流量。 |

這兩個端點只提供簡短狀態，適合部署平台、反向代理或外部監控系統輪詢，不提供詳細診斷資料。

---

## 2. 人工巡檢頁

管理員可直接開啟：

```text
/system-health
```

此頁不放入左側選單，必須先以 Cookie 登入，且目前使用者必須是管理員（`IsAdmin=true`）才可查看。

頁面內容包含：

- 整體健康百分比。
- 整體紅 / 黃 / 綠燈號。
- 每個檢查項目的紅 / 黃 / 綠燈號、權重、狀態、證據與失敗訊息。
- 頁尾顯示目前 NLog 今日日誌檔案的最後 100 筆原始紀錄。

---

## 3. 燈號與分數

整體健康分數採加權計算：

| 狀態 | 單項得分 | 燈號 |
|------|----------|------|
| Healthy | 取得完整權重 | 綠燈 |
| Degraded | 取得一半權重 | 黃燈 |
| Unhealthy | 不得分 | 紅燈 |

整體燈號門檻：

| 分數 | 燈號 | 說明 |
|------|------|------|
| `>= 90` | 綠燈 | 系統健康。 |
| `70-89` | 黃燈 | 系統可運作，但有項目需要關注。 |
| `< 70` | 紅燈 | 系統健康程度不足，應立即檢查失敗項目與日誌。 |

---

## 4. 第一版檢查項目

| 項目 | 主要證據 |
|------|----------|
| 網站 / 應用程式 | 環境、版本、啟動時間、運作時間。 |
| API | Controller action 數量、Swagger 啟用狀態。 |
| 資料庫 | Provider、連線狀態、待套用 migration 數量。 |
| 日誌 | NLog 目錄、今日 log 檔、最後 100 筆讀取狀態。 |
| 身分驗證 | Cookie scheme、JWT bearer、Issuer/Audience 是否設定、SigningKey 長度。 |
| 檔案系統 | Database、Download、Upload、ProjectFile、TaskFile、MeetingFile 目錄存在與可寫入。 |
| 主機資源 | Process working set、目前磁碟可用空間。 |
| 安全設定 | Swagger in Production、CORS origins、`Security.ReturnExceptionDetails`。 |

---

## 5. 日誌顯示注意事項

`/system-health` 頁尾會顯示最後 100 筆原始日誌紀錄，不會額外遮罩內容。此設計是為了讓管理員快速追查失敗項目，但也代表頁面可能顯示路徑、帳號或例外訊息。

部署時請確認：

- 只有可信任管理員可以登入此頁。
- NLog 不應寫入密碼、token、完整連線字串等 secret。
- 正式環境的日誌保存期限與存取權限符合組織政策。
