# 開發慣例與限制速查（AI／開發者必讀）

> 動手改本專案前先讀本文件。這裡集中列出設計慣例、不變量與踩雷點；違反標 ⚠️ 的項目通常會改壞功能或留下隱患。完整文件索引見 [`docs/README.md`](../README.md)。

---

## 1. 分層與依賴方向 ⚠️

依賴方向一律向上，**不得反向相依**：

```
MyProject.Web ──► MyProject.Business ──► MyProject.AccessDatas
       │                  │
       └──► MyProject.Dtos / MyProject.Models / MyProject.Share
```

- `MyProject.Share`、`MyProject.Models`、`MyProject.Dtos`：不得相依其他專案。
- UI（Razor）**不得直接 `using` `BackendDBContext`**，只能呼叫 Service / Repository。
- 細節見 [架構總覽](架構總覽.md)。

---

## 2. 模型變更要產生「雙資料庫」Migration ⚠️

本專案同時支援 SQLite（預設）與 SQL Server，兩者的 migration 分屬不同組件：

- SQLite：`src/MyProject/MyProject.AccessDatas/Migrations/`
- SQL Server：`src/MyProject/MyProject.AccessDatas.SqlServerMigrations/`（組件標記 `SqlServerMigrationAssemblyMarker.cs`）

**任何 Entity／欄位變更都要兩邊各產一次 migration**，否則切換 provider 後資料表不一致。指令見 [EFCore 指令備忘](../guides/EFCore.md)；切換說明見 [SQL Server 切換說明](../guides/SQL%20Server%20切換說明.md)。

---

## 3. EF 追蹤與編輯隔離 ⚠️

- 表格列被編輯前要先 `Clone()` 隔離，避免雙向綁定污染來源資料。
- Update / Delete 前以 `CleanTrackingHelper`（`src/MyProject/MyProject.Business/Helpers/CleanTrackingHelper.cs`）清除追蹤實體，避免 EF 追蹤衝突。
- 附件「資料表紀錄」由 EF Core Cascade 處理，但**實體檔案**仍須由 Service 層在刪除前後手動移除。見 [檔案上傳機制](../features/檔案上傳機制.md)。

---

## 4. Web API 回應一律包信封 ⚠️

- 任何 Controller 一律回傳 `ApiResult<T>`；分頁則包 `PagedResult<T>`（定義於 `MyProject.Dtos`）。
- UI 用 Cookie 認證（`MagicObjectHelper.CookieScheme`）、API 用 JWT Bearer，兩套 scheme 互不干擾。
- CRUD API 預設需要 Bearer token；login / refresh 允許匿名。未授權／禁止存取的回應也包成 `ApiResult`。
- 平行路由：保留 `/api/...`，新用戶端優先用 `/api/v1/...`。
- 見 [Web API 設計慣例](Web%20API%20設計慣例.md)、[API Versioning 策略](API%20Versioning%20策略.md)、[認證授權與權限機制](../security/認證授權與權限機制.md)。

---

## 4.1 紀錄分類/團隊標籤與團隊權控 ⚠️

- Project/MyTask/Meeting 的 `Categories`/`Teams` 為**多值標籤**，以換行分隔字串儲存；務必透過 `TagStringHelper`（`ToStored`/`ToList`）轉換，AutoMapper 已以 `ForMember` 處理 List↔字串。
- **團隊行級權控**只在 **Blazor 服務層**：清單 `GetAsync` 對非管理員套 `BuildTeamAccessPredicate`，單筆 `GetAsync(id)`／檔案下載以 `IsTeamAccessible` 守門。非管理員僅見「無團隊（公開）或與其角色團隊交集」的紀錄。透過 `IRecordAccessScopeProvider` 取得範圍。
- 使用者團隊來自角色 `RoleView.DefaultTeamsJson`，登入時載入 `CurrentUser.TeamList`（見 `AuthenticationStateHelper.Check`）。
- Web API（repository 路徑）目前**不**做行級過濾（與母專案一致）。

## 5. 權限與選單同步 ⚠️

側邊選單與權限樹以**位置索引**對應，三處需一致：

- `src/MyProject/MyProject.Web/Datas/Menu.json` — Sidebar 結構與權限節點。
- `RolePermissionService`（`src/MyProject/MyProject.Business/Services/Other/RolePermissionService.cs`）— 以 `GetRolePermissionAllNameToJson()` 產生 `RoleView.TabViewJson`（`List<List<string>>`）。
- `MagicObjectHelper`（`src/MyProject/MyProject.Share/Helpers/MagicObjectHelper.cs`）— 預設角色／開發者帳號等識別常數。

新增需要權限控管的頁面時，上述三處要同步加入對應節點，否則選單顯示或授權會錯位。

## 6. 命名差異提示 ⚠️

- DbSet 名稱維持 `MyTas` / `MyTasFile`（早期命名），Entity 已更名為 `MyTask`。新增 migration 時留意此差異，避免產生不必要的 rename。
- 命名對齊慣例：`XxxPage` / `XxxView` / `XxxService` / `XxxRepository` / `XxxAdapterModel` / `XxxDto`。
- 通知統一走 AntDesign 的 `NotificationService` / `MessageService`。

---

## 7. 啟動時的 Seed 行為

`Program.cs` 於 `app.Build()` 後會：套用 migration 或 `EnsureCreated()`、Seed 預設角色（`MagicObjectHelper.預設角色`，覆寫 `TabViewJson`）、Seed 開發者帳號（`MagicObjectHelper.開發者帳號`，密碼以 `PasswordHelper.GetPasswordSHA(salt, password)` 雜湊並重置 `IsAdmin=true`）。此為冪等行為。預設帳號種子值來自 `BootstrapSettings`。見 [資料模型與資料庫](資料模型與資料庫.md)、[密碼種類與儲存機制](../security/密碼種類與儲存機制.md)。

---

## 8. 版本與文件 ⚠️

- **單一版本來源**：`appsettings.json` 的 `SystemSettings.SystemInformation.SystemVersion`，每次完成異動一律將**最後一碼（Patch）+1**（例：`0.4.0 → 0.4.1`，不進位、不分異動性質；顯示格式 `Major.Minor.Patch (YYYY/MM/DD)`），並於 commit message 說明。
- **文件編碼**：`docs/` 下所有 `.md` 一律 **UTF-8 含 BOM**，CI 以 [`scripts/Test-DocsEncoding.ps1`](../../scripts/Test-DocsEncoding.ps1) 遞迴強制；繁體中文不得有亂碼。
- 異動若影響既有功能／API／資料表／設定／UI，須同步更新對應文件並更新索引。對照表見 [維護規範](../operations/維護規範.md)。
- 引用程式碼請用「相對路徑」（必要時加行號），勿臆測。

## 9. 上線前必查

正式部署前務必檢查 JWT `SigningKey`（不可沿用開發預設，否則啟動中止）、預設帳號密碼、`Swagger:EnabledInProduction`、CORS、`CacheSettings`（Redis 連線字串）。完整清單見 [正式部署與安全檢查清單](../operations/正式部署與安全檢查清單.md)。
