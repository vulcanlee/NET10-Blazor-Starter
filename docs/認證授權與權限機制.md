# 認證授權與權限機制

## 目的
本文件說明腳手架目前同時支援 Blazor UI Cookie 登入與 Web API JWT Bearer 認證。UI 與 API 使用不同 authentication scheme，避免互相干擾。

## Cookie 認證
Blazor UI 既有登入流程維持 Cookie authentication：
- scheme 使用 `MagicObjectHelper.CookieScheme`。
- Login path 為 `/Auths/Login`。
- Logout path 為 `/Auths/Logout`。
- Access denied path 為 `/Auths/Login`。

此機制主要服務 Razor/Blazor 頁面與既有後台使用者體驗。

## JWT Bearer 認證
Web API 使用 JWT Bearer：
- JWT 設定來源為 `appsettings.json` 的 `JwtSettings`。
- `JwtSettings` 會綁定到強型別 `MyProject.Web.Auth.JwtSettings`。
- Access token 用於 API 呼叫授權。
- Refresh token 採 signed JWT，不落庫。

`JwtSettings` 欄位：
- `Issuer`
- `Audience`
- `SigningKey`
- `AccessTokenMinutes`
- `RefreshTokenDays`
- `ClockSkewMinutes`

## Auth API
目前提供三個 API：
- `POST /api/Auth/login`：使用帳號密碼登入，成功回傳 access token 與 refresh token。
- `POST /api/Auth/refresh`：使用 refresh token 換發新 token。
- `GET /api/Auth/me`：使用 Bearer access token 取得目前使用者 DTO。

## API 授權規範
- Project/MyTask/Meeting CRUD API 預設全部需要 Bearer token。
- 登入與 refresh API 允許匿名。
- API 未授權與禁止存取回應也會包成 `ApiResult`。

## Refresh Token 限制
目前 refresh token 不落庫，因此有以下限制：
- 無法可靠撤銷單一 refresh token。
- 使用者登出後，已發出的 refresh token 仍可能在有效期內換發 token。
- 若要強制全部失效，只能輪替 `SigningKey` 或縮短有效期。
- 未來若需要可撤銷、裝置管理或風險控管，應改成 refresh token 落庫並記錄 jti、使用者、過期時間、撤銷時間與裝置資訊。

## 安全待辦
- [ ] 正式環境必須改用安全 secret 來源提供 `JwtSettings:SigningKey`。
- [ ] 評估正式環境是否停用或限制 Swagger UI。
- [ ] 建立預設帳號強制改密碼流程。
- [ ] 若需要可靠登出與撤銷能力，將 refresh token 改為落庫版本。
