# 正式部署與安全檢查清單

## 目標
- [x] 建立腳手架上線前必查項目，避免開發用設定被複製到正式環境。

## JWT 與認證
- [ ] 將 `JwtSettings:SigningKey` 改由 secret manager、環境變數或部署平台 secret 提供。
- [x] Production 啟動時若仍使用開發用 signing key，會直接中止啟動。
- [ ] 確認 signing key 長度、亂數強度與輪替流程。
- [ ] 確認 access token 與 refresh token 有效期符合系統風險等級。
- [ ] 記錄 refresh token 採 stateless signed JWT，無法可靠撤銷單一 token；若需要撤銷能力，需改成落庫或加入 token version 機制。
- [ ] 確認 `ApiResult.Exception` 正式環境完整回傳的資訊揭露風險已被接受，或另開需求改成依環境遮蔽。

## 預設帳號
- [ ] 部署前停用、刪除或透過 `BootstrapSettings` / 環境變數重新設定預設 `support` 帳號。
- [x] Production 啟動時若仍使用 `support` 預設密碼，會直接中止啟動。
- [ ] 若系統需要保留預設管理者，必須將 `BootstrapSettings:SupportPassword` 改由部署 secret 提供；首次啟動強制改密碼可另開功能化任務。
- [ ] 檢查文件、seed、測試資料中沒有可被正式環境沿用的密碼。

## Swagger 與 API
- [ ] 正式環境確認 Swagger UI 是否關閉，或限制在內網/VPN/授權後使用。
- [x] Production 必須明確設定 `Swagger:EnabledInProduction`，預設範本為 `false`。
- [ ] 確認所有 CRUD API 均需要 Bearer token。
- [ ] 確認 API validation、401、403、404、500 都維持 `ApiResult<T>` 格式。
- [ ] 確認檔案下載成功時維持原生 File stream，錯誤才回 `ApiResult<T>`。
- [ ] 確認新用戶端使用 `/api/v1/...`，既有 `/api/...` 僅作相容路由。

## 部署設定
- [ ] 設定 HTTPS、HSTS、反向代理 header 與正式網域。
- [ ] 設定 NLog 路徑與 log retention，避免敏感資料長期保留。
- [ ] 設定資料庫備份、migration 執行策略與 rollback 流程。
- [ ] 檢查 CORS、cookie、SameSite、upload/download 目錄權限。
- [ ] 確認 `/health/live` 與 `/health/ready` 已納入部署平台或監控系統。
- [ ] 確認 `/system-health` 只有管理員可存取，且頁尾最後 100 筆原始日誌不會暴露 secret。
