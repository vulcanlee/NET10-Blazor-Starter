# 記住我登入原理說明

## 文件目的

說明本專案在登入頁勾選「記住我」之後，為什麼下次重新開啟網站時可以不必重新做身分驗證，以及這個機制在本專案中是如何實作的。

## 結論

「記住我」的核心不是把密碼記在前端，也不是跳過登入檢查，而是：

1. 使用者第一次成功登入時，伺服器建立一張 **持久化的登入 Cookie**。
2. 瀏覽器把這張 Cookie 存起來，不只在目前分頁有效，關閉後重新開啟網站時也會再帶回來。
3. 之後每次請求網站時，瀏覽器都會自動把這張 Cookie 傳給伺服器。
4. 伺服器驗證 Cookie 合法後，就能還原目前使用者的登入身分。
5. 因此使用者不需要再次手動輸入帳號、密碼與驗證碼。

也就是說，「記住我」真正記住的是 **登入狀態**，不是帳號密碼本身。

---

## 本專案的程式流程

## 1. 系統先註冊 Cookie 驗證機制

檔案：`src/MyProject/MyProject.Web/Program.cs`

專案在啟動時有註冊 Cookie 驗證：

- `builder.Services.AddAuthentication(MagicObjectHelper.CookieScheme)`
- `.AddCookie(MagicObjectHelper.CookieScheme, options => { ... })`

目前設定重點：

- `options.LoginPath = "/Auths/Login";`
- `options.LogoutPath = "/Auths/Logout";`
- `options.AccessDeniedPath = "/Auths/Login";`
- `options.Cookie.IsEssential = true;`

對應程式位置：
- `MyProject.Web/Program.cs:101-108`

### 這段的作用

這表示系統採用 **Cookie Authentication** 做登入狀態管理。

當使用者登入成功後，系統會發出一張驗證 Cookie；之後的請求只要帶著這張 Cookie，系統就能知道這個人之前已經登入過。

其中：

- `LoginPath`：未登入時導到登入頁
- `LogoutPath`：登出時走的路徑
- `AccessDeniedPath`：權限不足時導回登入頁
- `Cookie.IsEssential = true`：把這張登入 Cookie 標記為必要 Cookie，避免被 Cookie 同意政策擋掉

---

## 2. 使用者在登入頁勾選「記住我」

檔案：`src/MyProject/MyProject.Web/Components/Auths/Login.razor`

登入頁面有一個 `RememberMe` 欄位，會綁定到後端的 `Input.RememberMe`。

這個欄位的意思是：

- `false`：只建立目前瀏覽器工作階段有效的登入 Cookie
- `true`：建立可跨瀏覽器重新開啟後仍存在的持久化 Cookie

---

## 3. 登入成功後，系統把 RememberMe 寫進 AuthenticationProperties

檔案：`src/MyProject/MyProject.Web/Components/Auths/Login.razor.cs`

登入成功後，程式會建立 `AuthenticationProperties`：

- `IsPersistent = Input.RememberMe`
- `RedirectUri = returnUrl`

對應程式位置：
- `MyProject.Web/Components/Auths/Login.razor.cs:111-115`

接著呼叫：

- `HttpContext.SignInAsync("CookieAuthenticationScheme", new ClaimsPrincipal(claimsIdentity), authProperties)`

對應程式位置：
- `MyProject.Web/Components/Auths/Login.razor.cs:117-123`

### 這裡是整個「記住我」的關鍵

`IsPersistent = Input.RememberMe` 代表：

- 如果沒有勾選記住我：`IsPersistent = false`
  - 產生的是 **Session Cookie**
  - 通常只在目前瀏覽器工作階段有效
  - 關閉瀏覽器後，下次再開啟通常就不在了

- 如果有勾選記住我：`IsPersistent = true`
  - 產生的是 **Persistent Cookie**
  - 瀏覽器會把 Cookie 保存到本機
  - 下次重新啟動瀏覽器或重新打開網站時，Cookie 仍可能存在
  - 因此不需要重新登入

簡單講：

> 勾選「記住我」後，ASP.NET Core 不是把登入狀態只放在記憶體，而是要求瀏覽器把驗證 Cookie 持久保存。

---

## 4. Cookie 裡面裝的是什麼

登入成功時，程式會先建立 Claims：

- `ClaimTypes.Role`
- `ClaimTypes.Name`
- `ClaimTypes.NameIdentifier`
- `ClaimTypes.Sid`

對應程式位置：
- `MyProject.Web/Components/Auths/Login.razor.cs:100-106`

這些 Claims 會組成 `ClaimsIdentity`，再由 Cookie 驗證機制序列化成驗證票證，最後寫進 Cookie。

### 重點

Cookie 內通常不是明文帳號密碼。

它保存的是一份經過 ASP.NET Core Cookie Authentication 處理過的 **驗證票證**，裡面包含使用者識別資訊與必要宣告。伺服器在後續請求中讀到這張 Cookie，就能重建 `ClaimsPrincipal`，知道目前是誰登入。

所以：

- 不需要再次輸入密碼
- 但也不是把密碼原封不動存在 Cookie 內

---

## 5. 下次重新開啟網站時，為什麼還知道你是誰

當使用者重新開啟網站：

1. 瀏覽器對網站送出新請求
2. 如果本機還保留那張持久化驗證 Cookie，就會自動附帶在請求中
3. ASP.NET Core 的 Cookie 驗證中介軟體會先讀取 Cookie
4. 如果 Cookie 有效，就把目前請求的 `HttpContext.User` 設成已登入使用者
5. Blazor / 元件層再透過認證狀態取得目前登入者資訊

在本專案中，後續檢查使用者身分的地方是：

檔案：`src/MyProject/MyProject.Business/Services/Other/AuthenticationStateHelper.cs`

### 這段實際在做什麼

`AuthenticationStateHelper.Check(...)` 會：

1. 透過 `authStateProvider.GetAuthenticationStateAsync()` 取得目前認證狀態
2. 檢查 `user.Identity.IsAuthenticated`
3. 從 Claims 中取出 `ClaimTypes.Sid`，並確認它是有效的使用者 Id
4. 依使用者 Id 再去資料庫抓完整使用者資料
5. 檢查使用者是否存在、是否停用、是否需要強制變更密碼
6. 初始化目前使用者權限與角色資訊
7. 把資料寫進 `currentUserService.CurrentUser`

`Check(...)` 目前會回傳 `AuthenticationCheckResult`：

- `Succeeded`：檢查成功，已建立目前使用者狀態與權限。
- `Unauthenticated`：目前 principal 未登入，系統會導向登出流程。
- `InvalidUser`：Sid 無效、使用者不存在、帳號停用、角色資料缺失或權限初始化失敗。
- `RequiresPasswordChange`：使用者需要先到 `/ChangePassword` 完成強制改密碼。

所以「重新進站後仍已登入」不是單靠前端記憶，而是：

- 瀏覽器保留 Cookie
- 伺服器驗證 Cookie
- 系統再從 Claims 與資料庫重建目前使用者狀態

---

## 6. 為什麼不需要重新驗證碼

驗證碼只在 **登入當下** 用來防止機器人或暴力嘗試。

一旦首次登入成功，系統就已經發出有效的驗證 Cookie。之後再次進站時，系統是根據 Cookie 來識別你，而不是重新走登入表單流程，所以也不需要重新輸入驗證碼。

換句話說：

- 驗證碼是登入前的防護
- Cookie 是登入後的身分憑證

兩者用途不同。

---

## 7. 什麼情況下即使勾選記住我，還是可能失效

即使有勾選「記住我」，也不代表永遠不會失效。常見原因如下：

### 7.1 Cookie 過期

持久化 Cookie 仍然有有效期限。

本專案目前沒有在 `AddCookie(...)` 裡顯式設定 `ExpireTimeSpan`，因此會採用 ASP.NET Core Cookie 驗證的預設行為。到期後仍然需要重新登入。

### 7.2 使用者主動登出

檔案：`src/MyProject/MyProject.Web/Components/Auths/Logout.razor.cs`

登出時會呼叫：

- `HttpContext.SignOutAsync(MagicObjectHelper.CookieScheme)`

對應程式位置：
- `MyProject.Web/Components/Auths/Logout.razor.cs:27`

這會刪除目前登入對應的驗證 Cookie，所以之後再進站就要重新登入。

### 7.3 瀏覽器清除 Cookie

如果使用者手動清除瀏覽器 Cookie，或使用隱私模式 / 無痕模式，持久化 Cookie 可能不會被保留。

### 7.4 伺服器無法接受原本的 Cookie

例如：

- 系統的 Cookie 保護金鑰變了
- 驗證票證失效
- Cookie 被竄改
- 站台網域或安全設定變更

這些情況都可能讓舊 Cookie 無法再被接受。

### 7.5 使用者資料狀態改變

在本專案中，登入後還會再檢查：

- 使用者是否存在
- `ClaimTypes.Sid` 是否存在且可解析成有效使用者 Id
- 使用者帳號是否仍為啟用狀態
- 是否需要變更密碼
- 是否具備角色資料
- 權限資料能否正常初始化

若其中任一檢查失敗，系統仍可能導向登出或要求重新處理；若回傳 `RequiresPasswordChange`，則會先導向 `/ChangePassword`，不會直接進入一般系統頁面。

## 8. 本專案裡「記住我」的完整事件順序

### 第一次登入時

1. 使用者輸入帳號、密碼、驗證碼
2. 使用者勾選「記住我」
3. `LoginUser()` 驗證帳密與驗證碼
4. 建立 Claims 與 `ClaimsIdentity`
5. 設定 `AuthenticationProperties.IsPersistent = true`
6. 呼叫 `HttpContext.SignInAsync(...)`
7. 伺服器回應驗證 Cookie
8. 瀏覽器把 Cookie 保存到本機

### 下次重新開啟網站時

1. 瀏覽器重新請求網站
2. 自動帶上之前保存的驗證 Cookie
3. 伺服器驗證 Cookie
4. `HttpContext.User` 被還原成已登入使用者
5. `AuthenticationStateHelper` 讀取 Claims 與資料庫資料
6. 系統建立目前使用者狀態與權限
7. 使用者直接進入系統，不必重新登入

## 9. 常見誤解

### 誤解一：記住我是把密碼存起來

不是。

一般情況下，記住我保存的是登入票證或驗證 Cookie，不是明文密碼。

### 誤解二：記住我代表永不登出

不是。

只要 Cookie 過期、被清除、被登出，或伺服器不再接受該 Cookie，就仍然需要重新登入。

### 誤解三：重新開網站不用登入，代表系統沒做驗證

不是。

系統仍然有做驗證，只是驗證依據從「帳密 + 驗證碼」改成「先前登入成功後簽發的驗證 Cookie」。

---

## 10. 與本次修正的關係

本專案先前有兩個與 RememberMe 相關的重點：

1. `RememberMe` 欄位本身要能正確送出表單值
2. 登入 Cookie 要能被正常保存

其中：

- `Login.razor` 已將 RememberMe 改為較適合 Blazor 表單繫結的 `InputCheckbox`
- `Program.cs` 已將 `options.Cookie.IsEssential = true` 加入 Cookie 設定，避免必要登入 Cookie 被 Cookie 同意機制擋下

這兩者一起作用後，RememberMe 才能正常完成：

- 表單送出成功
- `LoginUser()` 讀到 `Input.RememberMe = true`
- 系統建立持久化驗證 Cookie
- 下次重開網站時可還原登入狀態

---

## 11. 一句話總結

勾選「記住我」之後，系統會在登入成功時發出一張可持久保存的驗證 Cookie；瀏覽器下次再開網站時自動帶回這張 Cookie，伺服器驗證通過後就能還原你的登入身分，因此不需要重新做帳密與驗證碼驗證。
