# 腳手架新專案啟動流程

## 目標
- [x] 建立從本腳手架複製成新系統時的檢查清單，避免命名、設定、帳號與文件殘留舊專案資訊。

## 改名待辦
- [ ] 替換 solution、project、namespace 與 assembly 名稱，例如 `MyProject` 改成新系統代號。
- [ ] 檢查 `src/MyProject/MyProject.slnx` 內所有 project path 是否同步更新。
- [ ] 更新 `appsettings.json` 的 `SystemSettings:SystemInformation`，包含系統名稱、描述與版本。
- [ ] 更新 README、docs 與 CI workflow 顯示名稱。
- [ ] 檢查 UserSecretsId、NLog 路徑、資料庫檔案路徑、`BootstrapSettings` 與部署環境 secret。

## API 與 DTO 待辦
- [x] Web API request/response 使用 DTO，不直接暴露 Entity。
- [x] 所有 API 回應統一封裝為 `ApiResult<T>`。
- [ ] 新增業務模組時，同步建立 `CreateUpdateDto`、`Dto`、`SearchRequestDto` 與 mapping。

## 驗收標準
- [ ] `dotnet build src/MyProject/MyProject.slnx -v:minimal --no-incremental` 成功。
- [ ] `dotnet test src/MyProject/MyProject.Tests/MyProject.Tests.csproj -v:minimal` 成功。
- [ ] Swagger UI 可使用 Bearer token 呼叫受保護 API。
- [ ] 新專案文件不再殘留不應公開的開發帳號、JWT key 或舊系統名稱。

## 備註風險
- [ ] 若只改 UI 顯示名稱但未改 namespace，未來套件發布、log 分類與維運追蹤會混淆。
- [ ] 預設帳號目前仍是腳手架初始化流程的一部分，正式部署前必須依 release checklist 處理。
